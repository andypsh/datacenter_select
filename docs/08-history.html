주요 결정 (의사결정 로그)

결정 1: Vue 3 vs Streamlit (frontend)

선택: Vue 3.

근거: 슬라이더-지도 반응성, 디자인 자유도, 정적 호스팅 영구 무료, 제품·서비스 부문 정합성.

대안 검토: 다른 세션 산출물(산출물_사양.md 초안)에 Streamlit 적혀 있었음 → Vue로 갱신.

결정 2: 9개 지표 모두 구현 vs 5개로 시작

선택: 9개 전부 구현 (공모전 사양 정합).

근거: 컨셉_리포지셔닝.md의 평가지표 표를 100% 반영해야 심사위원에게 일관성 있는 작품이 됨.

결정 3: tier proxy vs V1 raw 파생

선택: 파생.

근거: 사용자가 "파생변수 만들었어?" 질문으로 proxy 한계 지적 → 진짜 raw에서 실측화. 결과 차이 큼 (예: 수원 통신 82→90, 광케이블 노드 인접도 반영).

결정 4: KOSIS 막힘 시 우회

선택: SW회사 raw × 기업규모별 표준 종사자 추정.

근거: Chrome MCP에서 kosis.kr 도메인 차단. SGIS 시도 → 생활업종(PC방·가구)이라 부적합. V1 raw에 기업규모 정보가 있어서 표준 종사자 곱하기로 추정 가능.

결정 5: docs/ 정적 HTML

선택: docs/ 폴더에 index + 9개 페이지.

근거: 사용자 요청 — "정식 서비스 아닌 docs용". Vue 앱과 분리, 빌드 없이 바로 열림.

버려진 시도

  • data.go.kr SGIS 생활업종 종사자수 순위 — 다운로드 성공했으나 IT 인력 지표로 부적합. 삭제.
  • data.go.kr 표준데이터 CSV 버튼 직접 클릭 — 처음엔 JS 클릭이 freeze. 좌표 클릭으로 우회 성공.
  • 모든 지표 한 번에 9개 확장 시도 — proxy 한계가 사용자에게 발각됨. 실데이터 파생으로 재작업.

학습된 교훈

1. "구현 완료" 표현 주의: proxy를 실데이터처럼 표현하면 안 됨. 명확히 "tier 기반 추정" 명시.
2. 폴더 통합은 git mv로: 326개 파일을 rename으로 추적해 히스토리 보존. 단순 mv는 add+rm으로 보임.
3. Vue HMR + Pinia state: store 구조 변경 시 페이지 강제 리로드 필요. 초기화 의존 코드는 load()에 집중.
4. ECharts 0×0 버그: v-show로 hidden 컨테이너에서 mount하면 canvas 0. v-if로 매번 mount.
5. 도메인 차단은 우회: KOSIS 막혔지만 data.go.kr 가능. 항상 alternative 출처 탐색.
